1. Front End 구현: React (화면 담당)

사용자가 보는 화면 만들기
버튼, 입력창, 카드, 목록, 페이지 구성
클릭/입력 같은 사용자 동작 처리
※ 없으면 생기는 문제 --> 화면이 없음, 사용자가 앱을 볼 수 없음. 로그인창, 버튼, 게시판 UI 등을 만들 수 없음

2. Front End 개발 실행: Vite (React 개발 도구)

npm run dev로 실행 (Node Package Manager)
React 프로젝트를 로컬에서 실행
개발 중 화면 미리보기
코드 수정 시 바로 반영
※ 없으면 생기는 문제 --> React 코드를 편하게 실행 및 확인인하기 어려움 (주방), 빌드/실행 환경을 직접 구성해야 함

3. Front End 배포: Vercel (화면을 인터넷에 올리는 곳)

만든 React 화면을 인터넷에 공개
다른 사람이 접속 가능한 주소 제공
배포 자동화, HTTPS, 도메인 연결 지원
※ 없으면 생기는 문제 --> 내 PC에서만 볼 수 있음 (Local host, Network Only) Network밖의 사용자 접속 불가

4. Back End 구현 (Node.js): (뒤에서 일하는 처리 담당)

로그인 처리
회원가입 처리
게시글 저장/수정/삭제
API 제공
프론트와 DB 사이의 중간 처리
※ 없으면 생기는 문제 --> 복잡한 로직 처리 불가로그인 검증, 권한 처리, 서버 로직 구현이 어려움, DB와 안전하게 연결하기 어려움

5. Back End 개발 실행 (Express): node index.js (백엔드 서버 켜기) 

Express 서버를 로컬에서 실행
API 테스트 가능
프론트가 서버에 요청 가능
※ 없으면 생기는 문제 --> 백엔드가 안 켜짐. /api/login, /api/posts 같은 기능이 동작 안 함

6. Back End 배포: Heroku (백엔드 서버를 인터넷에 올리는 곳)

Express 서버를 인터넷에서 계속 실행
외부에서 API 호출 가능하게 함
서버를 항상 켜진 상태로 운영
※ 없으면 생기는 문제 --> 백엔드를 내 PC에서만 돌려야 함. 내 컴퓨터 꺼지면 서버도 종료

7. DB 저장: MongoDB /Atlas (데이터 창고)

회원정보 저장
게시글 저장
댓글, 로그, 설정값 저장
서비스 데이터 보관
※ 없으면 생기는 문제 --> 저장이 안 됨. 새로고침하면 데이터 사라짐. 로그인 정보, 게시글, 사용자 기록 유지 불가

<Heroku vs Firebsae>

Heroku는 “백엔드를 배포하는 곳”
Firebase는 “백엔드 기능을 대신 제공하는 서비스”

- Heroku 쓰는 구조
프론트엔드(React)
Node.js/Express Backend서버
MongoDB
Heroku 배포
여기서 실제 백엔드 서버는 Node.js/Express이고,
Heroku는 그 서버를 올리는 곳입니다.

- Firebase 쓰는 구조
프론트엔드(React)
Firebase Auth
Firestore
Storage (DB)
여기서는 전통적인 의미의 서버를 덜 만들고,
Firebase 기능으로 많이 대체합니다.

<MongoDB vs MongoDB Atlas>

- Mongodb 사용
DB 서버를 직접 설치하고 관리해야 함
로컬 개발은 가능
운영/배포가 어려워짐

- Atlas(MongoDB)사용
MongoDB를 클라우드에서 쉽게 운영 가능
배포 연동이 쉬움
관리 부담이 줄어듦
